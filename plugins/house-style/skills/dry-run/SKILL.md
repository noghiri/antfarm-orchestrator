# Dry-Run Compliance

Any **script, tool, or program generated by an agent** that performs destructive, irreversible, or side-effecting operations MUST support a dry-run mode. This is a non-negotiable house standard.

## Scope: what this skill governs

This skill applies to **code you write** — scripts, CLI tools, automation programs, deployment utilities, migration scripts, and similar artifacts that the system produces as output.

Examples:
- A deployment script that pushes containers or modifies infrastructure
- A migration utility that alters a database schema
- A data-processing tool that overwrites files
- Any script that calls external APIs with mutating effects

## Scope: what this skill does NOT govern

The orchestrator's own tool calls (GitHub label changes, issue creation, branch creation, PR creation) are governed by **stage-level authorization**, not per-operation dry-run. The human grants authorization when approving a stage gate, and that authorization covers all orchestrator mutations for that stage. Do not add dry-run guards to orchestrator skill logic.

## What requires dry-run in generated code

- File deletion or overwrite
- Git push, force-push, branch deletion
- Calls to external APIs that mutate state (not read-only fetches)
- Shell commands that modify system state
- Database writes or schema changes

## How to implement dry-run

Before executing any operation in the list above, check whether dry-run mode is active:

```
if DRY_RUN:
    print(f"[dry-run] would execute: {describe_operation()}")
    return
```

The dry-run output must be specific enough that a human can verify correctness without running the real operation. Include: what would happen, to which resource, with what parameters.

## Dry-run must be the default

When building any script or tool that performs destructive operations, `--dry-run` (or equivalent) must be the default mode. The user must explicitly opt in to live execution with `--execute` or `--no-dry-run`.

## Dry-run is not optional compliance

Dry-run support is not a "nice to have." Omitting it from generated code is a spec violation and will be caught during peer review. A reviewer that notices missing dry-run support in a generated tool must block the work unit.
